Three routes:

1. `GET /` — Calls `getDocumentsByScope({ scope: 'global' })`. Returns `res.json(documents)`.

2. `POST /` — Uses `upload.single('file')`. Extracts `req.file`, determines `fileType` from MIME/extension. Calls `knowledgeService.processDocument({ userId: req.userId, scope: 'global', fileName: req.file.originalname, fileType, fileBuffer: req.file.buffer })`. The processDocument call is fired asynchronously (no await) so the response returns immediately with status 201 and the initial document record (status: 'processing'). Use a pattern like: insert the document record first with status processing, then fire processDocument in background. Actually, since processDocument already inserts + processes + updates status, fire it with `.catch()` to log errors. But we need the initial doc record to return. Solution: call processDocument but do NOT await — respond with 201 immediately after getting the initial record. Better approach: since processDocument returns the final doc, we need a two-step: (a) insert doc record via supabaseAdmin directly with status 'processing', return it with 201, (b) fire background processing. BUT knowledgeService.processDocument already does the insert. So instead: call processDocument, DO NOT await, return 202 with `{ message: 'Processing started', document_id: null }`. ACTUALLY simpler: just await processDocument fully and return the result. The pipeline is fast (< 30s per Phase 10 spec). Return 201 with the document. If it errors, processDocument already marks it as 'error' and throws, so catch and return 500.

Final approach (simpler, matches Phase 10 design): `await processDocument(params)` fully. Return 201 with the document (which will have status 'ready' on success). If processDocument throws, it already marks the doc as 'error' internally, so catch the error and return 500 with `{ error: message }`.

3. `DELETE /:id` — Validates `req.params.id` is present. Calls `deleteDocument(id, req.userId)`. If returns false, respond 404. If true, respond 204.
