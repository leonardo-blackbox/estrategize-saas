**7. Similarity search function (for Phase 13 RAG, but define now):**
```sql
CREATE OR REPLACE FUNCTION public.match_knowledge_chunks(
  query_embedding vector(1536),
  match_threshold float DEFAULT 0.7,
  match_count int DEFAULT 5,
  filter_scope text DEFAULT NULL,
  filter_consultancy_id uuid DEFAULT NULL
)
RETURNS TABLE (
  id uuid,
  document_id uuid,
  content text,
  token_count integer,
  metadata jsonb,
  similarity float
)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT
    kc.id,
    kc.document_id,
    kc.content,
    kc.token_count,
    kc.metadata,
    1 - (kc.embedding <=> query_embedding) AS similarity
  FROM public.knowledge_chunks kc
  JOIN public.knowledge_documents kd ON kd.id = kc.document_id
  WHERE kd.status = 'ready'
    AND (filter_scope IS NULL OR kd.scope = filter_scope)
    AND (filter_consultancy_id IS NULL OR kd.consultancy_id = filter_consultancy_id)
    AND 1 - (kc.embedding <=> query_embedding) > match_threshold
  ORDER BY kc.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;
```

**Step 2: Create `backend/src/services/embeddingService.ts`**

This service wraps OpenAI's text-embedding-3-small API. It accepts an array of text strings and returns an array of number arrays (embeddings).

```typescript
import OpenAI from 'openai';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

const EMBEDDING_MODEL = 'text-embedding-3-small';
const MAX_BATCH_SIZE = 100; // OpenAI limit per request

export async function generateEmbeddings(texts: string[]): Promise<number[][]> {
  const embeddings: number[][] = [];

  for (let i = 0; i < texts.length; i += MAX_BATCH_SIZE) {
    const batch = texts.slice(i, i + MAX_BATCH_SIZE);
    const response = await openai.embeddings.create({
      model: EMBEDDING_MODEL,
      input: batch,
    });
    for (const item of response.data) {
      embeddings.push(item.embedding);
    }
  }

  return embeddings;
}
```

Key details:
- Uses `text-embedding-3-small` model (1536 dimensions)
- Batches in groups of 100 (OpenAI limit)
- Returns `number[][]` matching input order

**Step 3: Create `backend/src/services/knowledgeService.ts`**

This is the main orchestration service. It has 3 public functions:

1. `processDocument(params)` — Full pipeline: parse file, chunk text, generate embeddings, store in DB
2. `deleteDocument(documentId, userId)` — Delete document and cascade-delete chunks
3. `getDocumentsByScope(params)` — List documents by scope (for future API routes)

Implementation details:

```typescript
import { createClient } from '@supabase/supabase-js';
import pdfParse from 'pdf-parse';
import { generateEmbeddings } from './embeddingService.js';

const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);

// --- Constants ---
const TARGET_CHUNK_TOKENS = 400;   // target chunk size
const MIN_CHUNK_TOKENS = 300;      // minimum viable chunk
const MAX_CHUNK_TOKENS = 500;      // hard cap
const OVERLAP_TOKENS = 50;         // overlap between chunks
const AVG_CHARS_PER_TOKEN = 4;     // rough approximation for splitting

// --- Types ---
interface ProcessDocumentParams {
  userId: string;
  scope: 'global' | 'consultancy';
  consultancyId?: string;
  fileName: string;
  fileType: 'pdf' | 'txt' | 'md';
  fileBuffer: Buffer;
}

interface Chunk {
  content: string;
  tokenCount: number;
  index: number;
}
```

**parseFile function:**
- For `pdf`: use `pdfParse(fileBuffer)` and return `result.text`
- For `txt` and `md`: return `fileBuffer.toString('utf-8')`

**chunkText function:**
Implements recursive character splitting:
1. Split text into paragraphs (by `\n\n`)
2. Accumulate paragraphs into chunks targeting `TARGET_CHUNK_TOKENS` (use `Math.ceil(text.length / AVG_CHARS_PER_TOKEN)` for token estimation)
3. When accumulated text exceeds `MAX_CHUNK_TOKENS`, finalize current chunk
4. Apply overlap: prepend last `OVERLAP_TOKENS * AVG_CHARS_PER_TOKEN` characters from previous chunk to next chunk
5. Discard chunks below `MIN_CHUNK_TOKENS` unless it's the only chunk
6. Return `Chunk[]` with content, estimated tokenCount, and sequential index

**processDocument function:**
1. Insert row into `knowledge_documents` with status `'processing'`
2. Parse file content via `parseFile()`
3. Split into chunks via `chunkText()`
4. Generate embeddings via `generateEmbeddings(chunks.map(c => c.content))`
5. Insert all chunks into `knowledge_chunks` with their embeddings using a batched approach (format embedding as string `[0.1,0.2,...]` for Supabase insert)
6. Update document: `status = 'ready'`, `chunk_count = chunks.length`
7. On any error: update document `status = 'error'`, `error_message = err.message`
8. Return the document record

**deleteDocument function:**
1. Verify document belongs to userId (SELECT with user_id filter)
2. Delete from `knowledge_documents` (CASCADE deletes chunks)
3. Return success/failure

**getDocumentsByScope function:**
Query `knowledge_documents` filtered by scope and optionally consultancy_id, ordered by `created_at DESC`.

Important implementation notes:
- Use `supabase.from('knowledge_chunks').insert()` for chunk storage. For the embedding vector, pass it as a string formatted like `[0.123,0.456,...]` — Supabase/PostgREST handles vector type conversion from string.
- The service uses `SUPABASE_SERVICE_ROLE_KEY` (bypasses RLS) since processing happens server-side.
- Wrap the entire processDocument in try/catch to ensure status is set to 'error' on failure.
- Log errors to console.error with `[knowledgeService]` prefix.
